# check-reachability スキル

ai-simulator の **会得式スキル**と**キャラ解放**について、「**実質的に到達不能なもの（詰み）が紛れていないか**」を点検する診断スキル。

## 大前提：何をする話なのか取り違えない

「スキル」という語が紛らわしい。**追加ではなく"点検"がこのスキル。**

- **このスキル＝点検／診断**。何も足さない。`scripts/audit-reachability.ts` を走らせて結果を読み解くだけ。読み取り専用で世界は進めない。
- **会得式スキルを"足す"** なら → `add-game-skill`。
- **登場人物（妖）を"足す"** なら → `add-character`。
- **行動（Action）を"足す"** なら → `add-action`。

ユーザーが「絶対取れないスキル／解放されないキャラがいないか確認して」と言ったら、ほぼ確実にこのスキル。

## 一発実行

```bash
bun run scripts/audit-reachability.ts          # 人が読むレポート
bun run scripts/audit-reachability.ts --json   # 機械可読（さらに掘るとき）
```

出力は `cmd > /tmp/audit.out 2>&1` でファイルに逃がしてから Read で読むと取りこぼさない（ライブ出力が空で返る既知バグ対策）。

まずこれを実行し、`🔴 総合判定` を中心にユーザーへ要約する。**実行する前にスクリプトの中身を再実装しようとしない**——既にこのスクリプトが静的・動的の両面を見るので、結果の解釈に集中する。

## 出力の読み方（🔴 / 🟡 / 🟢）

- **🔴 unreachable＝実質到達不能の疑い。** ここが本題。放置すると「永久に取れないスキル」「永久に出ないキャラ」になる。原因を切り分けて対処を提案する（下記）。
- **🟡 watch＝要観察。** まだ取れてないが到達経路はある／周回・データ不足で断定保留。原因と「あと何が要るか」を一言添える。
- **🟢 ok＝会得・解放の実績あり。** 触らない。

## 仕組み（なぜこの判定が信用できるか）

スクリプトは二段で見る。両方を理解しておくと 🔴/🟡 の意味をユーザーに正しく説明できる。

1. **静的チェック（コード論理・実データ非依存）**
   - 各 `skill.measure` を、全行動 × 極大ヒーロー（対人相手あり・餓死寸前・利他95・全報酬発生）でつついて、**どの行動でも 0 しか返さないスキル**を検出する。これが出たら行動名のタイポや死んだ条件＝**定義バグ**。
   - `loop` スコープなのに `threshold` が 1 周の最長（約30日＝30日目の大禍まで）を超えるもの＝**構造的に 1 周で届かない**。
   - 各 `CHARACTER_UNLOCKS.isUnlocked` を「全スキル会得・利他100・周1万」のマシマシ文脈で評価し、**それでも false なら論理的に充足不能**＝解放条件が壊れている。

2. **動的チェック（実ログ `data/world.db`）**
   - **専用監査ログ `skill_audit`**（毎 tick のスキル進捗・利他・解放ロスターの時系列スナップ。`server.ts` のワーカーが 1 tick ごとに 1 行書く）を最優先で使う。
   - 無ければ最新 campaign の `snapshot`（現在値）＋ `history`（周ごと要約）でフォールバック。
   - `career`（通算）スキルは snapshot の現在値がそのまま通算到達なので、**通算◯周で progress=0 → 達成条件を一度も満たしていない → 🔴**。
   - `loop`（毎周頭でリセット）スキルは「毎周どこまで届いたか」が要るため、**`skill_audit` に実在する周数が 3 周分未満なら断定せず 🟡（ログ不足）**。世界をもっと進めれば精度が上がる。

> 監査ログ `skill_audit` は後から導入した。**導入前のぶんは時系列が無い**ので、`loop` スキルの過去到達は追えない。「世界を何周か進めてから再実行すると精度が上がる」と案内してよい。

## 🔴 が出たときの掘り下げと対処の出し分け

🔴 は原因が2系統ある。**どちらかを切り分けてから提案する**（短絡的に「バグです」と言わない）。

- **(A) 定義バグ系**（静的チェック由来）: measure がどの行動でも 0／`isUnlocked` がマシマシでも false／loop で threshold 過大。
  → `src/domain/skills.ts`・`src/domain/characters.ts` の当該定義を開いて原因箇所を示し、修正案を出す。これは実データと無関係に直すべき。

- **(B) バランス系**（動的チェック由来・通算 progress=0 等）: 定義は健全だが、**主人公ハルがその行動を選ばない／その状態に届かない**ため実際には積み上がらない。
  例: `share`/`rest`/`guard` を AI がほぼ選ばず通算0、`sever_solitude` は利他70が必要だが観測ピーク28で頭打ち。
  → これは定義バグではなく「設計上は取れるが運用上は取れていない」状態。短絡修正しない。次のような選択肢をユーザーに提示して相談する:
    - 行動方針（`decide` のプロンプト／報酬設計）を見直してその行動が選ばれやすくする
    - 閾値や発火条件を実プレイに合わせて緩める
    - そもそも撤去・統合する
    - もっと周回を回して様子を見る（まだ偶然起きていないだけかもしれない）

キャラの 🔴 も同様で、「依存スキルが (B) で取れていない」なら、根は依存スキル側にある。連鎖を辿って真因を1つに絞る。

## ユーザーへの報告のしかた

- まず `🔴` を最優先で、**スキル名・キャラ名と一言の理由**を列挙する。0 件なら「詰みは無し」とはっきり言う。
- 各 🔴 について **(A)定義バグ / (B)バランス** の見立てを添える。確信が持てないときは該当定義を開いて確かめてから言う（推測で断定しない）。
- 🟡 は件数と代表例だけ簡潔に。全部を長々と並べない。
- 直し始める前に「どれを・どう直すか」をユーザーに確認する。点検が主目的なので、勝手に定義やバランスを書き換えない。
