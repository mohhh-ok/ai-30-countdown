# push スキル

ai-simulator の作業を `main` に安全に反映するための手順をまとめたスキル。
「とりあえず `git push`」ではなく、**レビュー → 自動修正 → ドキュメント判断 → commit → push** を
毎回同じ品質で回すのが目的。

## このプロジェクトの前提

`CLAUDE.md` に書かれている通り、これは個人プロジェクトで **`main` に直接 commit & push してよい**。
ブランチを切る必要はない。だからこのスキルは安心して main 上で完結させる。

ビルド/テストの自動コマンドはほぼ無い（`bun` プロジェクト、`package.json` の scripts は dev/start/sim のみ）。
型チェックだけ任意で `bunx tsc --noEmit` が使える。テストスイートは前提にしない。

## 全体の流れ

1. 変更を把握して stage する
2. feature-dev:code-reviewer でレビューする
3. 指摘を自動修正する（直せないものだけユーザーに確認）
4. ドキュメント更新の要否を判断し、必要なら更新する
5. commit してから push する

途中で「これは人間に判断してほしい」と感じたら止まって聞く。
それ以外は淡々と最後まで進めてよい。

## 1. 変更を把握して stage する

まず何が変わっているかを正確に掴む。

```bash
git status
git diff           # 未ステージの変更
git diff --staged  # ステージ済みの変更
git log --oneline -5
```

- 関係ある変更だけを `git add` する。生成物・一時ファイル・巨大バイナリ（`data/` の実行時生成物など）が
  混入していないか確認する。`.gitignore` で弾かれているべきものが untracked で出ていたら、コミットに含めず指摘する。
- 何を stage したのかを `git diff --staged --stat` で一度自分の目で確認してから次へ進む。

### 基本は「このセッションで自分が触った分」だけ

- push の対象は **原則として、このセッションで自分が編集したファイル/変更分のみ**。
  関係ない既存の未コミット変更（別作業・並行インスタンスのもの）まで巻き込んで commit しない。
- ただし **自分の変更と他の変更が同じファイル内で混ざっている（被っている）場合は、無理に切り分けなくてよい**。
  hunk 単位で機械的に分離しようとして事故るより、そのファイルごと素直に含めて構わない。
  混ざっていることに気づいたら、最後の報告で「○○は他の変更も一緒に入っている」と一言添える。
- push しようとした自分の変更が、並行作業によって既に commit / push 済みだった場合は、
  二重に何かをせず「すでに反映済み」と報告して終える。

### 画像は webp 限定（違反したら止まる）

このプロジェクトのキャラ絵・画像アセット（主に `assets/` 以下）は **`.webp` のみ許可**する。
これから commit に入る画像のうち `.webp` 以外（`.png` / `.jpg` / `.jpeg` / `.gif` / `.bmp` / `.tiff` など）が
1つでもあれば、**警告を出して push を中断**し、ユーザーに変換するか除外するか判断してもらう。勝手に進めない。

**合意済み例外（この2枚だけ非 webp を許可）:**

- `assets/og.jpg`（OGP シェアカード画像）。LinkedIn が WebP の og:image を
  公式非対応（JPG/PNG/GIF のみ）でプレビューを壊すため JPG を許可する（2026-06 合意）。
- `assets/favicon.png`（favicon = haru.webp の顔クロップ）。Safari が WebP の favicon に
  非対応のため PNG を許可する（2026-06 合意）。

チェックでこの2枚だけが出た場合は違反ではない。それ以外の非 webp は従来どおり止める。

stage 前後に必ずチェックする：

```bash
# これから commit される画像で webp 以外を検出
git diff --cached --name-only --diff-filter=ACM \
  | grep -iE '\.(png|jpe?g|gif|bmp|tiff?|webp)$' \
  | grep -ivE '\.webp$'
# 上の出力が空でなければ違反 → 止めて報告する
```

未ステージ/untracked にも紛れていないか `git status` の一覧で目視確認する。
（webp 変換は `docs/image-gen.md` の生成スクリプト側の責務。push スキルでは変換まではしない。）

## 2. feature-dev:code-reviewer でレビューする

stage した差分（= これから push する内容）を `feature-dev:code-reviewer` エージェントに渡してレビューさせる。
Agent ツールで `subagent_type: "feature-dev:code-reviewer"` を指定して起動する。

エージェントに渡すプロンプトには最低限これを含める：

- レビュー対象は「`git diff --staged` の差分（これから main に push する内容）」であること
- プロジェクトは bun + TypeScript + React の個人用シミュレータであること
- 重大度（critical / high / medium / low くらい）を付けて報告してほしいこと

レビューは **push をブロックするためではなく、自動修正のネタを集めるため**に回す、という気持ちで使う。

## 3. 指摘を自動修正する

レビュー結果が返ってきたら、自分で直せるものは直す。これがこのスキルの肝。

- **明確なバグ・ロジック誤り・規約違反・型エラー・分かりやすい改善**は、その場で Edit して直す。
- 直したら **対象差分をもう一度 code-reviewer に通す**（修正で新たな問題を持ち込んでいないか）。
  指摘が出続けるなら 2〜3 周まで。収束しなければそこで止めて状況を報告する。
- 次のものは自動修正せず、要約して**ユーザーに判断を仰ぐ**：
  - 仕様・意図が絡んで「どう直すのが正解か」がコードから判断できないもの
  - 直すと挙動が変わり、ユーザーの設計判断が必要なもの
  - 大きめのリファクタを伴うもの（push のついでにやるべきでない）

直したファイルは忘れず `git add` し直す。

任意で `bunx tsc --noEmit` を回して型が通ることを確認できるとなお良い（時間がかかりすぎる/設定が無い場合はスキップ可）。

## 4. ドキュメント更新の要否を判断する

差分の内容を見て、以下のドキュメントが**実態とズレないか**を必ず一通り照合する。
ズレるなら、その場で更新して同じ push に含める（更新後に `git add`）。

| ファイル | 更新が要るのはどんな変更か |
|---|---|
| `CLAUDE.md` | 運用方針・LLM 呼び出し方針・画像生成方針・Git 運用ルールが変わったとき |
| `README.md` | 機能の追加/削除、セットアップ手順・起動方法・概要・キャラ一覧が変わったとき |
| `docs/README.md` | docs/ 配下にファイルを追加/削除/リネームしたとき（インデックスを同期） |
| `docs/architecture.md` | ディレクトリ構成・ファイル追加/削除・技術スタック・スタイリング方針が変わったとき |
| `docs/game-rules.md` | 行動・場所・天候・災害・テンポ・回帰のルールや数値が変わったとき |
| `docs/characters.md` | キャラの追加/変更・解放条件・内部モデル・演出家・守護神の仕様が変わったとき |
| `docs/database.md` | テーブル追加/列変更・Drizzle 運用方針が変わったとき |
| `docs/api.md` | API エンドポイント・UIページの追加/削除/変更があったとき |
| `docs/env.md` | 環境変数の追加/削除/デフォルト値変更があったとき |
| `docs/llm-backend.md` | LLM バックエンドの追加/切替方式・課金まわりが変わったとき |
| `docs/cli.md` | CLI オプションの追加/変更・到達可能性アウディットの仕様が変わったとき |
| `docs/deploy.md` | デプロイ先・公開時の制約が変わったとき |
| `docs/secrets.md` | gitleaks 設定・CI パイプラインが変わったとき |
| `docs/image-gen.md` | 画像生成モデル・スクリプト・出力先・Web 組み込み状況が変わったとき |

判断の指針：

- **「このドキュメントだけ読んだ人が、今のコードを誤解しないか？」** を基準にする。誤解するなら直す。
- コメントの誤字直しや内部リファクタなど、ドキュメントに影響しない変更なら**何も触らない**（過剰更新しない）。
- 新機能を足したのに README に載っていない、コマンド名や環境変数が変わったのに記述が古い、などは典型的な「直すべき」ケース。
- 迷ったら「更新した／しなかった」と理由を最後の報告に一言添える。勝手に大量加筆はしない。

## 5. commit して push する

stage 内容（コード修正＋必要ならドキュメント更新）が固まったら commit する。

- ブランチは切らない。`main` のまま進める。
- commit message は変更内容を端的に表す日本語で書く（このリポジトリの既存ログは日本語短文スタイル）。
- commit message の末尾には必ず次の行を入れる：

  ```
  Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
  ```

```bash
git commit -m "$(cat <<'EOF'
<変更内容を表す日本語の1行サマリ>

Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
EOF
)"
git push
```

push 後に `git status` で working tree がきれいなことを確認する。

## 最後の報告

ユーザーには次を簡潔に伝える：

- push した commit（サマリと sha）
- レビューで出た指摘と、自動修正した内容（あれば）
- ユーザー判断に回した指摘（あれば）
- ドキュメントを更新したか／しなかったか、とその理由
