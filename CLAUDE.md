# プロジェクト方針メモ

## 作業を始める前に（最初に読む）
- **ユーザーの意図を正しく理解するため、着手前にまずドキュメントへ目を通すこと。**
  - `README.md`: 実装の全体像・LLM バックエンド切替・仕組み。
  - `docs/`: 個別トピックの知見（`image-gen.md` / `runpod-serverless.md` など）。
  - この `CLAUDE.md`: Git 運用・LLM 呼び出し・画像生成の方針。
- 指示が曖昧なときも、これらの方針に照らして意図を汲んでから動くこと。
- なお `plan.md` は初期の v1 仕様で**内容が古い**ため、原則として参照しない。

## ハマりどころ / 反省ログ（過去の誤りを繰り返さない）

### 観客ビューと楽屋ビューは別物。混同しない
- このアプリには2つの表示がある。**「観客に何が見えるか」を論じるときは必ずどちらの話か確定させること。**
  - **観客ビュー = `FrontStage.tsx`**（`.stage` / `.scene`）。配信で眺める表。**数値・囁き・気分・実り操作・演出意図(intent)は一切出さない。** 演出家由来で出るのは `director.narration` **だけ**。
  - **楽屋ビュー = `TickLog.tsx`**（`.log` / `.log-tick`）。開発・観察用の裏側。intent（`演出: …`）・囁き・実り操作の数値・LLM時間などはここにだけ出る。
- 既定で画面に出ているのが必ずしも `TickLog` とは限らない（`view-toggle` で切替）。**「観客向け表記」の話なら `FrontStage` を読む。**

### 画面挙動はライブで確認する。古いスクショを根拠にしない
- リポジトリ直下の `ui-*.png`（`ui-director.png` 等）は**コミット時点の古いスクショ**。現状の描画とズレうる。
- 実際に何が出ているかは **ライブで確認**する: `curl http://localhost:5566/api/state` か、Playwright で `localhost:5566` を開いて DOM を見る。

### 推測で設計を語る前に、該当コンポーネントのコードを開く
- 「演出家がラベルを付けている」等、**コードを見ずに想像で構造を断定しない**。見当違いの前提で `AskUserQuestion` を投げない。
- 着手前に `README.md` と該当 `src/` を読む（このルールは下の「作業を始める前に」と同じ。実際に破って遠回りした実績あり）。
- なお `plan.md` は古い（参照禁止）。`docs/` と `README.md` を正とする。

### ドメイン補足（上の議論の前提）
- `director`（ナレーション含む）は `engine.ts` で **tick 冒頭・キャラ行動前**に生成される（入力は過去と狙いのみ）。囁きにキャラが抗えるため、**幕開けナレが予告した展開が起きずに結果とズレうる**。観客ビューで地の文と結果が食い違って見えるのはこれが原因。

## Git 運用
- これは個人プロジェクト。他インスタンス（別の Claude/作業）との作業分離は気にしなくてよい。
- 作業ブランチを切って隔離する必要はなく、**`main` に直接コミット＆push してよい**。
- 「commit/push して」と言われたら、ブランチを作らず `main` でそのまま進めてよい。

## LLM 呼び出し方針
- **本番（公開運用）では `claude -p`（Claude Code CLI）を使わない。**
  - Claude Code / サブスクは「個人が対話的に開発する」用途が前提。シミュレーションを自動で回し続け、その出力を YouTube やサイトで公開する＝「Claude を組み込んだサービスの運営」に近く、サブスク経由 CLI の想定用途から外れる。
  - 公開フェーズの本番ループは **Anthropic API（従量課金・商用ターム）クライアント**に寄せる。レート制限・ポリシーの両面で安全。
- ローカルで自分が見て楽しむ／身内に見せる程度の開発・検証段階に限り、`claude -p` 運用でよい。
- 設計上、LLM 呼び出し層は **CLI / API を差し替えられる構造**にしておく。

## デプロイ／本番運用方針
- **デプロイ先は Railway を使う。**（マネージド PaaS・常駐プロセス可）
  - 理由1（常駐できる）: このアプリは **サーバ側ワーカーが自走で1日ずつ世界を進める**配信モデル（`server.ts` の `workerLoop` ／ `WORKER_INTERVAL_MS` の休憩式 = 1 tick 完了後に N ミリ秒休む）。Railway はプロセスを起きっぱなしにできるので、この自走ループが **そのまま動く**。
  - 理由2（移植不要）: bun ランタイムをそのまま動かせ、`Bun.serve`・`bun:sqlite` も無改造で動く。（Cloudflare Workers だと V8 isolate で常駐不可・`Bun.serve`/`bun:sqlite` 不可となり、Durable Object の `alarm()` + D1 への作り替えが必要になる。常駐前提の本アプリには過剰。）
  - 理由3（安い）: Hobby プランが月 $5 で、その $5 はリソース使用クレジットとして使える。本アプリは LLM の外部 API 待ち（I/O）が主で自前 CPU はほぼ遊ぶため、使用量が $5 内に収まりやすい。
  - 永続化（SQLite）はディスクが消えないよう Railway の **ボリューム**に `data/world.db` を置く。
  - 進行間隔は env `WORKER_INTERVAL_MS`（休憩式）で調整する。cron 式（実時間同期）は不要なので使わない。
- **公開時、状態を変える API（POST）は外部に出さない（いたずら防止）。**
  - 観るだけ画面。外から叩ける変更系（tick を進める／リセット／ワーカー起動停止）は **エンドポイントごと削除**し、公開するのは読み取り専用（`GET /api/state`・`GET /api/health`・キャラ絵）だけにする。
  - 進行はサーバ内部の自走ワーカーのみが行う（起動時に自動で回り、外部からは止められない）。UI も進行操作を持たず、`/api/state` を一定間隔でポーリングして最新状態を映すだけ。

## 画像生成方針
- **キャラ絵などの画像生成は OpenAI `gpt-image-2` を使う。**（`OPENAI_API_KEY` / 従量課金 API）
- ただし**背景透過が欲しいときは `gpt-image-1`**（gpt-image-2 は透過非対応）。
- 出力は `cwebp` で WebP に変換して `assets/` に保存（軽量化＋透過保持）。
- 生成スクリプト・モデルの使い分け・知見は `docs/image-gen.md` を参照。
